"Yummy Planner" Readme

Short Description

This CS50W Capstone Project "Yummy Planner" is a Platform in which users can add and share recipes, plan out meals in a calender and add the ingredients they need to cook those meals in a shopping list.

How to run

Once all files are downloaded

Install virtual environment with python 3.10.4 installed and run 

env/bin/activate

Now...

cd to yummy

then run

python3 manage.py makemigrations

then

python3 manage.py migrate

then create a superuser

python3 manage.py createsuperuser

Follow the prompts to create the user.

then

python3 manage.py runserver

then open 127.0.0.1:8000 in Webbrowser. Now you can login with the superuser, or register an new user.

What's contained in the files I created (besides the initial Django files)?

The project contains one Django application called planner. Within the planner file structure the template folder was created with the "planner" folder for the apps html templates. The "registration" folder was created to contain the registration form for new users, and the login form. This was required by Django's built in authentication "django.contrib.auth". Furthermore I added a models.py file as a means to create Django ModelForms, which made it easier and quicker to generate HTML forms. Lastly there is a folder, where the recipe images uploaded by users are stored.

Distinctiveness and Complexity

Why I created the Application

After my first attempted Capstone project idea turned out to be too complex and unfinishable, I returned to the drawing board.

I came up with the idea improve on tasks that take up quite a bit of time in my life:

  • deciding what to cook
  • shopping the ingredients
  • and cooking (new) meals.

Having a repertoire of meals can be quite helpful, and having a digital record of that repertoire, even more so. As a means to facilitate meal planning, a calendar would be helpful, to which meals from the repertoire could be added quickly and easily. Besides needing to prepare when the meal is to be cooked, a shopping list generated from the recipes will help when buying all the ingredients. The app should need to allow users to individually select ingredients from a recipe or add all ingredients from that recipe. Additionally users need to be able to add custom elements to the shopping list.

Implementing these functions makes my app distinctive from previous projects in this course, since, besides being an original idea, they include the implementation of Django functionalities not covered in CS50W. The use of DateTime objects, regular expressions, multiple related models, model forms, file upload, deciding the complete code architecture, using mainly custom CSS, plus figuring out how to use Djano's built-in authentication should satisfy the complexity requirements. In the next section, all implemented functions will be described to further support my case.

Overview of implemented Functions

Authentification

  • Users are able to: Register, Login, Logout (in previous Projects this was preimplemented. In this app I had to figure it out using external resources.)

Add a Recipe

  • Users are able to add new recipes filling out the following form fields: name, description, upload an image, preparation time, cooking time, serving size, ingredients list and instructions.
  • Ingredients and their respective amounts are stored seperately, as to ensure that ingredients are only stored once. (Metric and short form e.g. "tsp" only)

My Recipes and Recipe Pages

  • users can see their own recipes in an overview
  • Each recipe is displayed in its own page with all necessary information
  • Ingredients from the recipe can be added indidually or completly to the shopping list
  • Recipes can be added to the meal calendar
    • If a recipe is added to the calendar, the recipe page includes the date and mealtime for when it is planned. Mealtimes are either breakfast, lunch or dinner.
    • If a planned meal is in the past, the user can see the history of that recipe (hidden by default) within the recipe page.

Browse Recipes

  • Here all recipes from all users are displayed so that they may be browse and added to the personal meal, planning calendar

Meal calendar

  • Here in typical calendar view shows current month
  • Navigation buttons can let users change which month is viewed
  • In each day, the meals planned for that day and the attributed mealtimes are shown. They act as a link to the respective recipe page.
  • Recipes in the past can be marked as cooked or can be deleted from the history (in case the user did not cook the meal and wishes to remove it from the history)
  • If a meal is marked as cooked, it is shown as crossed out (line-though) in the calendar and the history shown in the recipe page
  • The coloured day in the calendar indicates the current date

Shopping list

  • The shopping list displays all individual ingredients added from recipes
  • If an ingredient was added multiple times, the ingredient only appears once in the list and the amounts from each time the ingredient was added are listed behind the ingredient
  • These amounts behind each ingredient item are a link to the recipe they originated from
  • Users can cross off the individual items while shopping, and once the shopping is complete, they can save the changes to the shopping list (in other words, remove the items that were crossed off)
  • Additional items which are not included in a recipe can be added to and checked off in a separate list displayed and next to the recipe ingredients
